# Multi-Dimensional Analysis: Executive Presentation Restructuring

## Topic 1: Narrative Frameworks Comparison

### Framework Comparison Matrix

| Dimension | Minto SCQ | Duarte "What Is/Could Be" | Raskin 5-Part |
|-----------|-----------|---------------------------|---------------|
| **Origin** | McKinsey consulting (1980s) | Presentation design firm (2010) | Strategic narrative consulting (2016) |
| **Best for** | Analytical/recommendation presentations | Inspirational/change management | Sales/funding/approval presentations |
| **Structure** | Linear: S→C→Q→A | Oscillating: Reality↔Vision | Progressive: Change→Stakes→Promise→Solution→Proof |
| **Opens with** | Shared context (Situation) | Current reality ("What Is") | External market shift |
| **Emotional arc** | Logical tension → resolution | Emotional contrast (hope ↔ frustration) | Urgency → aspiration → confidence |
| **Audience role** | Decision-maker asked to approve | Hero who can create change | Winner who must act now |
| **Data handling** | Pyramid of supporting evidence | Visual/cinematic slides + slidedocs | Features as "magic gifts" solving obstacles |
| **Ideal length** | Any (scales to memo or 50-slide deck) | 15-30 minutes | 10-20 slides |
| **Executive fit** | ⭐⭐⭐⭐⭐ (designed for it) | ⭐⭐⭐⭐ (needs restraint on emotion) | ⭐⭐⭐⭐⭐ (designed for buy-in) |

### Ideal Slide Count Research

| Source | Recommendation | Context |
|--------|---------------|---------|
| Guy Kawasaki "10/20/30 Rule" | 10 slides, 20 minutes, 30pt font minimum | VC pitch presentations |
| PechaKucha format | 20 slides × 20 seconds = 6:40 total | Creative presentations |
| Nancy Duarte (HBR) | 1 slide per 2-3 minutes of speaking | Executive presentations |
| McKinsey standard | 1 key message per slide, appendix for detail | Consulting presentations |
| Cognitive load research | 8-10 slides for 20 min; 10-15 for 30 min | Research-based |

**Consensus for 20-30 minute executive presentation: 10-12 slides** with detailed appendix/leave-behind.

## Topic 2: Pilot Program Framing Analysis

### The Perception Problem

Presenting "12.5% complete" to executives triggers several cognitive biases:

| Bias | How It Manifests | How to Counter |
|------|-----------------|----------------|
| **Anchoring** | Executives anchor on "12.5%" and see a barely-started project | Anchor on "Phase 1 complete" instead |
| **Status quo bias** | "If it's only 12.5% done, maybe we should wait" | Frame delay as losing competitive advantage |
| **Risk aversion** | Incomplete = risky investment | Position as de-risked (lighthouse proven) |
| **Sunk cost fallacy** | Could work FOR you — "we've invested, let's see it through" | Better: frame as rational next step with proven returns |

### The "Lighthouse" Reframe — Multi-Dimensional Analysis

**Security dimension:** A phased rollout is actually MORE secure — each phase incorporates lessons learned, reducing risk of enterprise-wide failure.

**Cost dimension:** The lighthouse model front-loads the expensive learning (building the framework). Subsequent rollouts are cheaper per-unit because the playbook exists. Show the decreasing marginal cost curve.

**Implementation complexity:** Phase 1 was the hardest (inventing the model). Phase 2+ is replication, which is fundamentally simpler. This should be shown visually.

**Stability:** One live implementation provides real-world stability data that theoretical plans cannot. "We don't just think this will work — we've seen it work."

### Framing Strategies Ranked by Executive Impact

1. **🥇 "Prove and Scale"** — Most compelling. Shows completed proof, asks for scale authorization.
2. **🥈 "Lighthouse/Flagship"** — Strong metaphor. Implies visibility and guidance for others.
3. **🥉 "Franchise Model"** — Relatable for multi-brand context. Every franchise starts with one location.
4. **4th: "Phase-Gated"** — Project management language. Less inspiring but familiar.
5. **5th: "MVP/Iteration"** — Tech startup language. May feel immature for executive context.

## Topic 3: Anti-Pattern Analysis

### Anti-Pattern #1: Information Redundancy

**The Problem:** Multiple slides covering overlapping content from different angles.

**In the Delta Crown deck currently:**
- Slide 5 ("The Delta Crown Model™") and Slide 5b ("SharePoint Hub & Spoke Architecture") may overlap — same model described architecturally vs. conceptually
- Slide 6 ("Operational Efficiency Gains") and Slide 8 ("Cost Optimization Strategy") likely share metrics
- Slide 10 ("Scalability Roadmap") and Slide 7 ("Implementation Playbook") may cover similar timeline content

**Cognitive load impact:** Redundant information doesn't reinforce — it confuses. The audience wonders "didn't we already cover this?" which breaks their mental model.

**Fix:** Consolidate related content. Each slide should pass the "Is this genuinely new information?" test.

### Anti-Pattern #2: The "Feature Dump"

**The Problem:** Listing every capability, feature, and technical detail as if comprehensiveness equals persuasiveness.

**Symptoms in executive presentations:**
- Data tables with 10+ rows on a single slide
- Technical architecture diagrams with every component labeled
- Lists of ALL operational procedures rather than representative examples
- Multiple metrics presented without a single clear takeaway

**Why executives reject feature dumps:**
- They don't have context to evaluate individual features
- Each feature creates a new node in working memory (exceeds Miller's 7±2)
- Features without context feel like undifferentiated list items
- Executives think in outcomes, not features

**The Raskin antidote:** Each feature is a "magic gift" that solves a specific obstacle on the path to the Promised Land. This gives each feature CONTEXT and PURPOSE.

### Anti-Pattern #3: Vision-Detail Tension

**The Problem:** Oscillating between high-level vision ("we'll transform multi-brand operations!") and granular technical detail ("here's the SharePoint column taxonomy") without consistent altitude.

**The "altitude model" for executive presentations:**

```
Altitude Level     Appropriate For
──────────────────────────────────────
30,000 ft         Board of Directors (vision, strategy, ROI)
20,000 ft         C-Suite (strategic framework, key metrics, decision points)  ← TARGET
10,000 ft         VP/Director (implementation approach, resource needs)
5,000 ft          Manager (detailed plans, workflows, timelines)
Ground level      Individual contributor (technical specs, code, configurations)
```

**Fix:** Maintain 20,000 ft altitude throughout the main deck. ALL ground-level detail goes to appendix/slidedoc.

### Anti-Pattern #4: Excessive Slide Density

**Research-backed density guidelines:**

| Element | Current Risk | Evidence-Based Max | Source |
|---------|-------------|-------------------|--------|
| Words per slide | Often 80-120+ | 25-40 words | Sweller cognitive load theory |
| Bullet points | 6-8+ per slide | 3-5 | Miller's 7±2 limit |
| Charts per slide | 2-3 | 1 (with max 3-4 data series) | Split-attention effect |
| Font sizes | 4-5+ | 2-3 hierarchical levels | Visual processing research |
| Key messages | 2-3 per slide | 1 per slide | McKinsey standard |

### Anti-Pattern #5: Missing the "So What?"

**The Problem:** Presenting information without explicitly connecting it to what the executive should DO with that information.

**Example:**
- ❌ "Our SharePoint architecture uses a Hub & Spoke model with 5 connected site collections"
- ✅ "Our Hub & Spoke architecture means adding a new brand takes 2 weeks instead of 6 months — that's the difference between seizing a market opportunity and missing it"

**The "So What?" test for every slide:**
1. If an executive asks "So what?" after this slide, can I immediately answer?
2. Does this slide move the audience closer to the decision I'm seeking?
3. Would removing this slide weaken my argument?

If the answer to any is "no," the slide should be cut or merged.
